# 2026-06-08 工作日志

## 开发了 Gitea CI/CD 前后端分离方案

用户需求：开发一个 Gitea 版本的 CI，支持 Java 和 Go，前后端分离，使用 Kaniko 完成镜像构建和推送，不需要代码扫描/测试/安全。

### 产出文件（`/Users/leo/WorkBuddy/Claw/gitea-ci-demo/`）

**三个 Gitea Actions 工作流（`.gitea/workflows/`）：**
- `ci-go-backend.yml` — 监听 `backend-go/**` 变更，Go 编译 + Kaniko 构建推送
- `ci-java-backend.yml` — 监听 `backend-java/**` 变更，Maven 编译 + Kaniko 构建推送
- `ci-frontend.yml` — 监听 `frontend/**` 变更，npm 构建 + Kaniko 构建推送

**示例应用代码：**
- `backend-go/` — Go 1.22 HTTP API，多阶段 Dockerfile
- `backend-java/` — Spring Boot 3.2 + Java 17，多阶段 Dockerfile
- `frontend/` — React 18 + Vite 5 + Nginx，多阶段 Dockerfile

**公用脚本：**
- `scripts/kaniko-build.sh` — 通用 Kaniko 构建脚本，处理 registry 认证和 executor 调用

**部署文件：**
- `docker-compose.yml` — 本地 Gitea + act_runner 编排
- `DEPLOY.md` — 含完整部署步骤、Runner 注册、Secrets 配置、故障排查
- `README.md` — 项目说明

### 技术要点
- Kaniko 认证通过 `/kaniko/.docker/config.json` 注入 base64 auth
- 镜像标签策略：branch-{sha} + latest，tag 推送时直接用 tag 名
- Runner 需要预装 `/kaniko/executor`，提供了自定义 Runner 镜像方案
- 触发条件用 `paths:` 过滤，只触发变更目录对应的工作流
